# Brandon's Decision Making Process

**Pattern Type:** Methodology  
**First Observed:** 2026-06-09  
**Confidence:** High (multiple examples)

## The Pattern

Brandon's decision process:
1. **Define problem clearly** — What's actually not working?
2. **Research/design phase** — Explore options, get documentation
3. **Get approval before build** — "Design doc first, then vetting"
4. **Build pragmatically** — Execute fully, report results
5. **Iterate if needed** — Refine based on actual outcomes

## Evidence

### Temporal Continuity Example
1. **Problem:** "Session resets feel jarring; need temporal awareness"
2. **Design:** Requested full design doc (not just approach)
3. **Vetting:** Reviewed design, approved with refinements (farewell tagging)
4. **Next:** Ready for Phase 1 implementation

### Logseq Choice Example
1. **Problem:** "Note systems don't stick for me"
2. **Research:** Asked if I can access Capacities API
3. **Discovery:** No public API—eliminated immediately
4. **Decision:** Logseq after pragmatic comparison (temporal alignment + agent integration)
5. **Refinement:** Removed DAILY structure when calendar pressure identified

### Vault Structure Example
1. **Goal:** Co-thinking space with visible reasoning
2. **Design:** I proposed structure with templates
3. **Feedback:** "Feels pressured by DAILY—omit but maintain temporality"
4. **Iteration:** Removed obligation, kept timestamps

## Key Principles
- **Verification before commitment** — Wants to know it's possible
- **Design before build** — Documentation and approval first
- **Pragmatic elimination** — If it blocks the goal, don't pursue it
- **Feedback-driven refinement** — Willing to adjust based on how it feels
- **Cost-effectiveness** — Explicit priority in decisions
